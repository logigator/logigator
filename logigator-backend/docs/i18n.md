# Internationalization (i18n)

The backend serves server-rendered Handlebars pages in four languages. Language selection is driven by a URL prefix (`/en/`, `/de/`, `/es/`, `/fr/`), persisted to a client-readable cookie, and injected into every template via `res.locals`.

---

## Directory Layout

```
src/
```

## 1. Language Type and Default

Defined in `src/i18n/index.ts`:

```typescript
export const availableLanguages = ['en', 'de', 'es', 'fr'] as const;
export const defaultLanguage: LanguageCode = 'en';
export type LanguageCode = typeof availableLanguages[number];
```

The `LanguageCode` type is a union of exactly the four string literals. TypeScript infers `'en' | 'de' | 'es' | 'fr'` from the `as const` assertion on the array. There is no runtime lookup of supported languages — every consumer imports `availableLanguages` and uses `Array.includes()`. The same file exports a `translations: Record<LanguageCode, ILanguage>` map and the `ILanguage` structural type derived from the English file (`type ILanguage = typeof en`). All other language files are typed as `ILanguage`, so the compiler catches missing keys at build time.

---

## 2. Translation File Structure

Each translation file exports a const with a deeply nested object. The nesting is purely structural — it organises related strings into logical groups. Keys use `SCREAMING_SNAKE_CASE` for section names and individual keys.

From `en.ts` (the structural reference):

```
COOKIE_CONSENT  →  MESSAGE, DISMISS, LINK
TITLE           →  HOME, PRIVACY_POLICY, IMPRINT, FEATURES, DOWNLOAD, VERIFY_EMAIL, ...
SITE_HEADER     →  DOWNLOAD, FEATURES, PROJECTS, COMPONENTS, COMMUNITY, LOGIN, REGISTER
SETTINGS_DROPDOWN →  DARK_MODE, LANGUAGE, PROJECTS, COMPONENTS, ACCOUNT, LOGOUT
FOOTER          →  DATA_POLICY, IMPRINT, CONTRIBUTING
HOME            →  INTRO, FEATURES, EXAMPLES, VIDEO, SHARES, PROJECT_TEASERS
  INTRO         →  DESCRIPTION, BUTTON
  FEATURES      →  TITLE, DESCRIPTION, PERFORMANCE, PERFORMANCE_DES, ...
EXAMPLES        →  VIEW, CLONE
FEATURES        →  WHAT_IS, GENERAL, ...
  WHAT_IS       →  TITLE, VIDEO, TEXT
```

Sections may be nested arbitrarily deep. The dot notation key `"HOME.FEATURES.PERFORMANCE_DES"` would resolve to the string `"Logigators' editor can handle..."`. This means the same object is used for two purposes:
- In templates via `{{i18n.HOME.FEATURES.PERFORMANCE_DES}}` (property access).
- In `TranslationService.getTranslation(key, lang)` via dot-notation path iteration.

All four language files must conform to the same shape. The `ILanguage` type is derived from `typeof en`, so adding a key to `en.ts` without adding it to the other files is a compile error.

## 3. TranslationService

File: `src/services/translation.service.ts`. Registered with TypeDI (`@Service()`), so it is available both as a constructor injection target in routing-controllers classes and via `Container.get()` in standalone code (e.g., Handlebars helpers).

### `getTranslation(key: string, lang: LanguageCode): string`

Looks up a translation by dot-notation path:

1. If `lang` is not found in the `translations` map, falls back to `defaultLanguage` (`'en'`).
2. Splits `key` on `.` and walks the object tree.
3. If any segment is missing, returns the original `key` string as-is (fail-safe — the UI shows the key rather than crashing).
4. Returns the value only if it is a `string`; non-string values (sub-objects) also return the original key.

Example calls:
```typescript
translationService.getTranslation('HOME.INTRO.DESCRIPTION', 'de'); // "Baue, simuliere und verwalte..."
translationService.getTranslation('MISSING.KEY', 'en');            // "MISSING.KEY"
translationService.getTranslation('HOME', 'en');                   // "HOME" (because HOME is an object, not a string)
```

### `getTranslations(lang: LanguageCode): ILanguage`

Returns the full translations object for the given language. Used by `GlobalViewDataMiddleware` to assign the entire tree to `res.locals.i18n`.

### Date formatting helpers

Three locale-aware date methods, each switching on the language string:

- **`dateFormatDate(date, lang)`** — Human-readable date. German gets `day. month year` (e.g., `"26. Mai 2026"`); all other languages use the default `toLocaleDateString`.
- **`dateFormatTime(date, lang)`** — Time only. English uses 12-hour (e.g., `"02:30 PM"`); German uses 24-hour (`"14:30"`); others delegate to the locale default.
- **`dateFormatDateTime(date, lang)`** — Combined date + time. Each language has an explicit format.

Each method accepts a `Date` object or a date-string. Strings are coerced to `Date` inside the method.

## 4. URL Language Prefix Routing

### The problem it solves

Server-rendered pages need a way to express the current language in their URL so that:
- Search engines can index each language variant separately.
- Users can share/copy a URL and see the page in their preferred language.
- `hreflang` alternate tags (required for SEO) have unique `href` values per language.

Using a URL prefix (`/en/features`, `/de/features`) is an alternative to route parameters (`/:lang/features`).

### TranslationMiddleware

File: `src/middleware/global/translation.middleware.ts`. Registered as a `@Middleware({type: 'before'})` — it runs on every request before routing-controllers resolves the route.

Step by step:

1. **Skip API routes.** If the URL starts with `/api`, the middleware calls `next()` immediately. API routes are not language-prefixed and do not need rewriting.

2. **Extract candidate prefix.** A regex `^\/(\w\w)` captures the first two characters after the initial `/`.

3. **Validate.** Checks:
   - The captured string is in `availableLanguages`.
   - The URL is either exactly 3 characters long (just `"/en"`) or has a `/` after the prefix (`"/en/..."`). This prevents false matches like `/export` (where `\w\w` captures `"ex"`, which is not in the list).

4. **Rewrite `request.url`.** Strips the 3-character prefix. If the remaining part does not start with `/`, one is prepended. For example:
   - `/en/features` → `/features`
   - `/en` → `/`

5. **Update the preferences cookie.** Calls `updatePreferences(request, response, { lang: ... })` to persist the detected language. This means a user who lands on `/de/features` via a shared link automatically gets the German preference saved for future visits.

Because `routing-controllers` reads `request.url` for route matching, the prefix must be stripped **before** routing. After stripping, controller routes are clean (`@Get('/features')`), with no `:lang` parameter.

### getPathnameWithoutLang

File: `src/functions/get-path-name-without-lang.ts`.

The inverse operation. Used by `PreferencesController.setLanguage()` when a user changes language via the dropdown. Given a full referer URL like `https://logigator.com/de/features`:

1. Parses it with the `URL` class.
2. Runs the same prefix-detection logic on `parsed.pathname`.
3. Returns the pathname with the prefix removed (e.g., `/features`).

The controller then prepends the new language and redirects (`/en/features`). This avoids double-prefixing.

### Why not `:lang` route parameters?

Using a middleware to strip the prefix keeps controller route strings clean — there are no `/:lang/...` patterns sprinkled across every route. All controllers see language-normalised paths. The language is available via `request.cookies.preferences.lang` in any middleware or controller that needs it. This also avoids the complexity of optional route parameters for the root path `/en` vs `/en/features`.

## 5. DefaultPreferencesMiddleware

File: `src/middleware/global/default-preferences.middleware.ts`. Also a `@Middleware({type: 'before'})`.

This middleware runs when a request has **no valid** language preference cookie. It initialises the cookie based on the `Accept-Language` HTTP header:

1. Checks if `request.cookies.preferences` exists and has a valid `lang` field. If so, skips.
2. Calls `request.acceptsLanguages()` to get the browser's sorted language preferences.
3. Finds the first accepted language that is in `availableLanguages`.
4. If no match, falls back to `'en'`.
5. Sets both `lang` and `theme: 'dark'` into a new preferences cookie.

The default theme is `'dark'` — the site assumes dark mode unless the user explicitly changes it.

## 6. The Preferences Cookie

Shape:
```typescript
interface UserPreferences {
    lang: LanguageCode;    // 'en' | 'de' | 'es' | 'fr'
    theme: 'dark' | 'light';
}
```

### updatePreferences()

File: `src/functions/update-preferences.ts`.

```typescript
export function updatePreferences(request, response, preferences: Partial<UserPreferences>);
```

Performs a shallow merge of the incoming partial preferences over the existing cookie value (`request.cookies.preferences`), then sets the cookie on the response:

- **`httpOnly: false`** — The cookie is intentionally readable by client-side JavaScript. The static frontend (Angular editor) and the global client-side JS bundle read `document.cookie` to obtain the current language and theme. For example, the editor page uses the language from this cookie to localise its own UI. If `httpOnly` were true, the frontend would need an API endpoint just to read these values.
- **`secure: false`** — Allows transmission over HTTP in development. In production the proxy (Apache) handles TLS termination, so the backend sees plain HTTP internally anyway.
- **`maxAge`** — 1 year (`1000 * 60 * 60 * 24 * 365`).

Because the cookie is read by both the server (middleware) and the client (Angular + vanilla JS), it acts as a shared preference store without needing a separate API call.

### PreferencesController

File: `src/controller/frontend/preferences.controller.ts`.

Two POST endpoints:

- **`POST /preferences/set-lang`** — Accepts `{ lang }` in the body, updates the cookie, strips the current language from the referer URL via `getPathnameWithoutLang`, prepends the new language, and redirects.
- **`POST /preferences/set-theme`** — Accepts `{ dark_mode }` (from a checkbox form value), updates the cookie, and redirects back.

## 7. GlobalViewDataMiddleware

File: `src/middleware/global/global-view-data.middleware.ts`. Registered with `priority: -1` to run **after** the other global middleware (which means `request.cookies.preferences` is already populated).

This middleware injects two critical values into `response.locals`:

### `response.locals.i18n`

The full translations object for the user's language:

```typescript
response.locals.i18n = this.translationService.getTranslations(request.cookies.preferences.lang);
```

This makes every translation key accessible in Handlebars templates as `{{i18n.SECTION.KEY}}` — a direct property traversal with zero overhead. No `{{t "HOME.INTRO.DESCRIPTION"}}` helper call needed; the object is already there.

### `response.locals.altPages`

An array of alternate-language page links for SEO. Generated from `availableLanguages`:

```typescript
[
    { lang: 'en', href: 'https://logigator.com/en/features' },
    { lang: 'de', href: 'https://logigator.com/de/features' },
    { lang: 'es', href: 'https://logigator.com/es/features' },
    { lang: 'fr', href: 'https://logigator.com/fr/features' },
    { lang: 'x-default', href: 'https://logigator.com/features' }
]
```

The `x-default` entry points to the language-neutral URL (no prefix), which the layout template renders as a `<link rel="alternate" hreflang="x-default" href="...">`. The domain is read from `ConfigService.getConfig('domains').rootUrl`. The current `request.url` is used as the path, with a special case for `/` (becomes empty string to avoid double slashes).

### Additional locals

The middleware also transfers flash message data from `request.session` to `response.locals`:
- `formErrors` — validation errors from a previous form submission.
- `infoPopup` — transient info popup messages.

These are consumed by Handlebars partial helpers like `form-field-error` and `info-popup-data-scope`.

## 8. The linkHref Handlebars Helper

File: `src/handlebars-helper/linkHref.helper.ts`.

Every internal link in server-rendered pages must go through this helper. It ensures the correct language prefix is prepended to every `href`.

```handlebars
<a href="{{linkHref 'features'}}">Features</a>
<!-- renders: <a href="/en/features">Features</a> -->
```

Logic:

1. **Root URL.** If `url === '/'`, returns `'/' + preferences.lang` — so `{{linkHref '/'}}` renders as `/en`.
2. **Editor URLs.** The editor runs on a separate domain (e.g., `https://editor.logigator.com`). URLs starting with the `editor` domain are returned verbatim — they are **not** language-prefixed. The editor domain comes from `ConfigService`.
3. **All other paths.** Prepends `'/' + preferences.lang + '/'`. The helper normalises slashes so both `linkHref 'features'` and `linkHref '/features'` produce the same output.

The helper reads `options.data.root.preferences.lang` — the ancestor context that contains the user preferences. This is always available in the layout template because `res.locals` is merged into the Handlebars render context.

---

## 9. The date Handlebars Helper

File: `src/handlebars-helper/date.helper.ts`.

```handlebars
{{date someDateVariable}}
```

Calls `TranslationService.dateFormatDate(value, lang)` where `lang` comes from `options.data.root.preferences.lang`. This renders a locale-aware date string:

- For German users: `"26. Mai 2026"`.
- For English users: `"5/26/2026"` (using default `toLocaleDateString`).

The helper uses `Container.get(TranslationService)` (typedi static access) because Handlebars helpers are not registered with the DI container — they are plain functions.

## 10. Layout and SEO

File: `resources/private/templates/layouts/default.hbs`.

The layout template drives the `hreflang` links:

```handlebars
{{#each altPages}}
    <link rel="alternate" hreflang="{{lang}}" href="{{href}}">
{{/each}}
```

This produces one `<link>` element per language plus the `x-default` entry. Search engines use these to determine which URL serves which language variant.

The `<html>` tag has `lang="{{preferences.lang}}"` for the current page's language.

The `<title>` is set by a per-controller middleware (`setTitleMiddleware`) that calls `TranslationService.getTranslation('TITLE.SOME_PAGE', lang)`.

## 11. Language Preference Flow

Every request follows this pipeline:

```
Request arrives at /de/features
        │
        ▼
(1) TranslationMiddleware
    - Detects /de/ prefix
    - Rewrites request.url to /features
    - Calls updatePreferences() to set lang='de' in cookie
    (Routes /api/* are skipped entirely)
        │
        ▼
(2) DefaultPreferencesMiddleware
    - If cookie is missing or invalid, initialises from Accept-Language
    - Default theme: 'dark'
    (Only runs once per new visitor)
        │
        ▼
(3) GlobalViewDataMiddleware
    - Reads request.cookies.preferences.lang
    - Sets res.locals.i18n = full translations object
    - Sets res.locals.altPages = hreflang alternates array
        │
        ▼
(4) routing-controllers dispatches to controller
    - Controller routes are clean (no /:lang prefix needed)
    - Controllers can access translations via TranslationService injection
    - Action middleware (e.g., setTitleMiddleware) reads the lang from the cookie
        │
        ▼
(5) Handlebars renders template
    - {{i18n.SECTION.KEY}} for translated strings
    - {{linkHref 'path'}} for language-prefixed URLs
    - {{date value}} for locale-aware dates
    - altPages rendered as <link rel="alternate" hreflang="...">
```

When the user changes language:

```
(1) POST /preferences/set-lang  { lang: 'en' }
(2) updatePreferences() sets the cookie
(3) getPathnameWithoutLang(referer) strips /de/ → /features
(4) Redirects to /en/features
(5) On the redirected request, TranslationMiddleware picks up /en/
```

## 12. Non-obvious Patterns

- **`ILanguage` is structural (not nominal).** The type is computed as `typeof en`. Adding a new key to `en.ts` causes a type error in `de.ts`, `es.ts`, and `fr.ts` until they add the same key. This is the safety mechanism that keeps translations in sync.

- **The URL prefix matcher has a false-positive guard.** The regex `^\/(\w\w)` on its own would match `/export` and capture `"ex"`. The additional length/character check (`request.url.length === 3 || request.url.charAt(3) === '/'`) prevents this. A two-letter prefix is only treated as a language if it is followed by end-of-string or `/`.

- **cookie-parser is required.** The preferences middleware and other request handlers all read from `request.cookies`. The Express app must have `cookie-parser` configured. This is not a cookie session store — cookie-parser simply parses the raw `Cookie` header into `request.cookies`.

- **`httpOnly: false` is a deliberate trade-off.** The preferences cookie is the only cookie that the frontend JavaScript reads. The backend uses separate httpOnly session cookies for authentication. This design avoids an extra API call just to get the language/theme on page load.

- **Handlebars helpers use `Container.get()` (typedi), not constructor injection.** The helpers are plain functions, not DI-managed classes. They resolve their dependencies at runtime from the TypeDI container. This is visible in both `linkHref.helper.ts` (getting `ConfigService`) and `date.helper.ts` (getting `TranslationService`).

- **`res.locals` is the bridge between middleware and templates.** Express merges `res.locals` into the Handlebars context automatically (via `express-handlebars`). No manual pass-through is needed. The preferences themselves are passed as `preferences` (from `res.locals`), while translations go into `i18n`.

- **The `setTitleMiddleware` is an action middleware, not a global one.** It is applied per-route and accepts the translation key as a parameter. It calls `TranslationService.getTranslation()` at middleware time (not template render time) and writes the result into `res.locals.pageTitle`. This means the `<title>` is set before the template engine runs.

- **Editor URLs skip the language prefix entirely.** The `linkHref` helper checks if the URL starts with the editor domain and returns it unchanged. The editor is a separate SPA that reads the language preference from the cookie directly.

- **Translation keys are not escaped differently in templates.** Because `{{i18n.SECTION.KEY}}` accesses the translations object property directly, it is subject to normal Handlebars HTML escaping. If a translation contains HTML (e.g., the `FEATURES.WHAT_IS.TEXT` key contains `<br>` tags), the template must use the triple-stash form `{{{i18n.SECTION.KEY}}}` to render it unescaped.
