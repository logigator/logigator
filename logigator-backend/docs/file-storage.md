# File Storage

## Overview

Logigator stores file-type data (circuit JSON blobs, preview images, and profile pictures) on the **filesystem**, not in MySQL. The database holds only metadata — a UUID filename, MIME type, and MD5 hash — while the actual bytes live under `resources/`. This keeps the database lightweight, enables efficient static serving of public assets via Express, and avoids bloating backups with binary data.

The system is built around a single abstract base entity, `PersistedResource`, that all file-like entities extend. Lifecycle hooks (`@AfterInsert`, `@BeforeUpdate`, `@BeforeRemove`) keep the filesystem in sync with the database automatically.

---

## Why the Filesystem?

Three categories of data are stored as files rather than database columns:

1. **Circuit JSON blobs (ProjectFile, ComponentFile).** A project or component serialises to a JSON string containing the full circuit graph — placed components, wire routes, connection points, and configuration. This JSON can be hundreds of kilobytes for a complex circuit. Storing it on the filesystem as a `.json` file avoids large TEXT columns and makes the data directly debuggable (you can open the file and inspect it).

2. **Preview images (ProjectPreviewDark, ProjectPreviewLight, ComponentPreviewDark, ComponentPreviewLight).** These are generated PNG renderings of circuits, uploaded by the editor. They are binary, potentially large per user, and served to unauthenticated visitors in the community gallery. Serving them statically via Express (rather than streaming from the database on every request) is cheaper and simpler.

3. **Profile pictures (ProfilePicture).** Uploaded avatar images. Same rationale as previews: binary, cacheable, served statically.

The unifying pattern: each entity has a one-to-one relationship with its parent (Project, Component, or User) and a `cascade: true` setting on the parent, so the file lifecycle follows the parent lifecycle automatically.

## The `PersistedResource` Abstract Entity

**File:** `src/database/entities/persisted-resource.entity.ts`

Every file in the system extends this abstract class. It is decorated with `@Exclude({toPlainOnly: true})` from `class-transformer`, so its internal columns are excluded from serialized API responses unless explicitly exposed.

### Database Columns

| Column | Type | Description |
|---|---|---|
| `id` | `uuid` PK | Primary key, auto-generated UUID. |
| `filename` | `uuid` (unique) | The filesystem name (without extension). Generated by TypeORM's `@Generated('uuid')` on insert. The extension is derived dynamically from `mimeType`. |
| `mimeType` | `varchar` | MIME type of the file (e.g. `application/json`, `image/png`, `image/jpeg`). Exposed to API consumers via `@Expose()`. |
| `hash` | `char(32)` | MD5 hex digest of the file content. Defaults to all zeros (`00000000000000000000000000000000`). Exposed to API consumers via `@Expose()`. |

### Transient Properties

| Property | Default | Description |
|---|---|---|
| `_fileContent` | `undefined` | In-memory buffer cache. Populated on first read by `getFileContent()`. |
| `_dirty` | `false` | Tracks whether the in-memory content differs from what is on disk and in the DB. |
| `_cacheable` | `false` | Flag that subclasses override. When `true`, updates generate a new UUID filename for HTTP cache-busting. |
| `_path` | `'persisted'` | Relative path segment (from `resources/`) that subclasses override. |

### Lifecycle Hooks

#### `@AfterInsert()` — `createFile()`

```typescript
@AfterInsert()
private async createFile() {
    await fs.writeFile(this.filePath, this._fileContent ?? Buffer.alloc(0));
    this._dirty = false;
}
```

Immediately after the entity row is inserted into the database, the file is written to disk at the resolved `filePath`. If no content was set (e.g., `_fileContent` is undefined), an empty buffer is written.

#### `@BeforeUpdate()` — `updateFile()`

```typescript
@BeforeUpdate()
private async updateFile(): Promise<void> {
    if (this._dirty) {
        if (this._cacheable) {
            await this.deleteFile();
            this._filename = uuid();
        }
        await fs.writeFile(this.filePath, this._fileContent);
        this._dirty = false;
    }
}
```

Before the entity row is updated in the database, the file is written to disk — but **only if `_dirty` is true**. This is the MD5-based change detection guard (see below). When `_cacheable` is true, the old file is deleted first and a new UUID filename is generated, which changes the URL and forces browser/CDN caches to fetch the new version.

#### `@BeforeRemove()` — `deleteFile()`

```typescript
@BeforeRemove()
private async deleteFile() {
    try {
        await fs.unlink(this.filePath);
    } catch (e) {
        if (e.code !== 'ENOENT')
            throw e;
    }
}
```

Before the entity row is deleted, the file on disk is removed. If the file does not already exist (`ENOENT`), the error is silently swallowed.

### Core Methods

#### `setFileContent(content: Buffer | string)`

Sets new content on the entity. Accepts either a `Buffer` or a string (which is converted to `Buffer.from()`).

The method computes the MD5 hash of the new content and compares it to the stored `_hash`:

```typescript
this._dirty = this._hash !== hash;
this._hash = hash;
```

This means:
- If the hash matches the stored value, `_dirty` stays `false` and the `@BeforeUpdate` hook **skips the filesystem write entirely**.
- If the hash differs, `_dirty` is set to `true`, so the next `save()` will write to disk.

This avoids unnecessary disk I/O when saving an entity whose content has not actually changed (e.g., re-saving a project with the same circuit JSON).

#### `getFileContent(): Promise<Buffer>`

Reads the file from disk into `_fileContent` (if not already cached in memory), then returns the buffer.

```typescript
public async getFileContent(): Promise<Buffer> {
    if (!this._fileContent)
        this._fileContent = await fs.readFile(this.filePath);
    return this._fileContent;
}
```

The in-memory cache persists for the lifetime of the entity instance. If you need to force a re-read from disk, you must replace the entity instance (e.g., by re-fetching from the repository).

#### `get filePath(): string`

Resolves the absolute filesystem path:

```typescript
public get filePath(): string {
    return path.join(
        __dirname, '..', '..', '..', 'resources',
        this._path,
        this._filename + '.' + mime.getExtension(this.mimeType)
    );
}
```

The extension is derived from `mimeType` using the `mime` npm package (e.g., `application/json` => `.json`, `image/png` => `.png`).

#### `get publicUrl(): string`

Generates a URL path suitable for use in HTML templates or API responses:

```typescript
public get publicUrl(): string {
    return `/${this._path.replace(/^public\//, '')}/${this._filename}.${mime.getExtension(this.mimeType)}`;
}
```

The `public/` prefix is stripped so the resulting path matches the Express static mount point. For example, a `ProfilePicture` with `_path = 'public/profile'` and filename `abc123` of type `image/png` produces `"/profile/abc123.png"`. This is served by Express from `resources/public/profile/abc123.png`.

This property is decorated with `@Expose()` so it appears in JSON API responses. Subclasses that should **not** expose a `publicUrl` (e.g., `ProjectFile`, `ComponentFile`) override it with `@Exclude()` to suppress it.

## The `_cacheable = true` Pattern

Preview images and profile pictures set `_cacheable = true`. When `_cacheable` is true and content is updated:

1. The existing file on disk is deleted.
2. A new UUID is generated and assigned to `_filename`.
3. The new content is written under the new filename.
4. The entity is saved to the database with the new filename, so the old filename is effectively abandoned.

This guarantees that the URL changes on every content update, which forces browsers and CDNs to fetch the new image rather than serving a stale cached version. Without this pattern, a user who updates their profile picture would see the old image until their cache expired.

The trade-off is orphaned files on disk. Old profile picture and preview files are never cleaned up automatically. They accumulate in `resources/public/profile/` and `resources/public/preview/{dark,light}/`.

Non-cacheable entities (`_cacheable = false`, the default) like `ProjectFile` and `ComponentFile` overwrite the same file in place. This is acceptable because:
- Circuit JSON is fetched by the authenticated editor, not by anonymous browsers, so cache-busting is handled differently (the editor knows the project has changed).
- The JSON is not served directly from the `resources/public/` tree — it goes through the API.
- Overwriting avoids orphan accumulation for frequently-saved documents.

## Concrete Subclasses and Their Paths

### JSON Blobs (Private — not served statically)

| Entity | `_path` | Extension | Parent | Purpose |
|---|---|---|---|---|
| `ProjectFile` | `private/projects` | `.json` | Project | Serialized circuit graph for a project |
| `ComponentFile` | `private/components` | `.json` | Component | Serialized circuit graph for a saved component |

Both classes hardcode `mimeType = 'application/json'` and override `publicUrl` with `@Exclude()` to prevent accidental exposure through the API.

### Preview Images (Public — served statically, cacheable)

| Entity | `_path` | Extension | Parent | Purpose |
|---|---|---|---|---|
| `ProjectPreviewDark` | `public/preview/dark` | `.png` | Project | Dark-theme thumbnail |
| `ProjectPreviewLight` | `public/preview/light` | `.png` | Project | Light-theme thumbnail |
| `ComponentPreviewDark` | `public/preview/dark` | `.png` | Component | Dark-theme thumbnail |
| `ComponentPreviewLight` | `public/preview/light` | `.png` | Component | Light-theme thumbnail |

All four hardcode `mimeType = 'image/png'` and set `_cacheable = true`.

### Profile Pictures (Public — served statically, cacheable)

| Entity | `_path` | Extension | Parent | Purpose |
|---|---|---|---|---|
| `ProfilePicture` | `public/profile` | `image/*` (dynamic) | User | Uploaded avatar |

`ProfilePicture` does not hardcode `mimeType` — it is set dynamically from the upload's `content-type` header. It sets `_cacheable = true`.

## The `resources/` Directory Layout

```
resources/
├── public/                          # Served statically by Express at /
│   ├── assets/                      # Static SVGs, brand logos, default images
│   ├── css/                         # Compiled SCSS (layouts/ + views/)
│   ├── js/                          # Compiled JS (global-es2015, global-es5, view scripts)
│   ├── cookieconsent/               # Cookie consent library assets
│   ├── preview/
│   │   ├── dark/                    # Dark-theme preview PNGs (cacheable, generated)
│   │   └── light/                   # Light-theme preview PNGs (cacheable, generated)
│   ├── profile/                     # Uploaded profile picture PNGs (cacheable, generated)
│   ├── persisted/                   # Legacy — previously used for file-backed entities
│   ├── vendor/                      # Third-party static assets (CodeMirror, etc.)
│   ├── sitemap.xml
│   ├── robots.txt
│   ├── browserconfig.xml
│   └── site.webmanifest
│
└── private/                         # NOT served by Express
    ├── components/                  # Component circuit JSON files (.json)
    ├── projects/                    # Project circuit JSON files (.json)
    ├── templates/                   # Handlebars templates (layouts/, partials/, views/, standalone/)
    ├── scss/                        # SCSS source files (compiled to public/css/)
    └── js/                          # JS source files (compiled to public/js/)
```

Express mounts the static file server in `main.ts`:

```typescript
app.use(expressStatic(path.join(configService.projectRootPath, 'resources', 'public'), {
    // ...
}));
```

Only `resources/public/` is served. The `resources/private/` tree is never accessible via HTTP — it contains circuit data, templates, and source files that are consumed only by the server process.

## File Lifecycle Through CRUD

### Create

1. A new entity instance is created (e.g., `new ProjectFile()` or obtained from `ProjectFileRepository.create()`).
2. Content is set via `setFileContent(buffer)` — this computes the MD5 hash and marks `_dirty`.
3. `mimeType` is set (hardcoded for JSON/PNG subclasses, or from the upload header for ProfilePicture).
4. The parent entity (Project, Component, or User) is saved with `cascade: true`, or the file entity is saved directly.
5. TypeORM fires `@AfterInsert()` -> `createFile()` writes the buffer to disk at the resolved `filePath`.
6. The UUID filename, MIME type, and MD5 hash are now stored in the database row.

### Read

1. The entity is fetched from the database (as part of a parent relation or directly).
2. `getFileContent()` is called.
3. If `_fileContent` is already cached in memory, it is returned immediately.
4. Otherwise, the file is read from disk via `fs.readFile()` and cached.
5. For public images, the `publicUrl` property is available via the API response without reading the file at all.

### Update

1. The entity is fetched from the database.
2. `setFileContent(newBuffer)` is called — MD5 is computed and compared to the stored hash.
3. If the content differs, `_dirty` is set to `true`.
4. The entity is saved to the database.
5. If `_dirty` is `true`:
   - For cacheable entities: old file is deleted, a new UUID is generated.
   - The new content is written to disk.
6. `_dirty` is reset to `false`.

**Important:** If the new content has the same MD5 hash as the existing content (e.g., saving the same circuit JSON twice), `_dirty` stays `false` and no filesystem write occurs. This is the primary benefit of the MD5 hash column — it prevents unnecessary disk I/O on no-op saves.

### Delete

1. The entity is deleted from the database (either directly or because its parent was deleted with `cascade`).
2. TypeORM fires `@BeforeRemove()` -> `deleteFile()` deletes the file from disk via `fs.unlink()`.
3. If the file does not exist (ENOENT), the error is silently swallowed — this is important for cacheable entities where the filename may have been updated and the old file deleted previously.

## Preview Image Upload Flow

Both the project and component controllers expose a POST endpoint for preview images:

```
POST /api/project/:projectId/preview
POST /api/component/:componentId/preview
```

The flow:

1. The frontend sends **two files** in a single multipart request (dark and light theme previews).
2. The `getUploadedFileOptions(2)` function configures **multer** with `memoryStorage()` (files are kept in memory as buffers, never written to a temp directory) and a `fileSize` limit of 1 MiB per file.
3. The controller receives the files as buffers via the `@UploadedFiles('previews', ...)` decorator.
4. The controller checks if preview entities already exist on the parent — if not, it creates new ones.
5. `setFileContent(buffer)` is called for each preview entity, which computes the MD5 hash.
6. The parent is saved, which cascades to the preview entities.
7. The `@BeforeUpdate` hook writes the PNG buffers to disk (with cache-busting via `_cacheable = true`).

```typescript
// From project.controller.ts
@Post('/:projectId/preview')
public async updatePreviews(
    @Param('projectId') projectId: string,
    @CurrentUser() user: User,
    @UploadedFiles('previews', {options: getUploadedFileOptions(2), required: true}) images: any
) {
    // ...
    if (!project.previewDark)
        project.previewDark = new ProjectPreviewDark();
    if (!project.previewLight)
        project.previewLight = new ProjectPreviewLight();

    project.previewDark.setFileContent(images[0].buffer);
    project.previewLight.setFileContent(images[1].buffer);
    // save cascades to preview entities
}
```

The same pattern applies to the component controller for ComponentPreviewDark/ComponentPreviewLight.

## Profile Picture Import

The `ProfilePictureRepository` provides a utility method for importing a profile picture from a remote URL:

```typescript
public async importFromUrl(url: string): Promise<ProfilePicture> {
    const resp = await fetch(url);
    const buffer = await resp.buffer();

    const resource = this.create();
    resource.setFileContent(buffer);
    resource.mimeType = resp.headers.get('content-type');

    return resource;
}
```

This is used during OAuth registration (Google/Twitter) to copy the user's existing avatar. The fetched buffer goes through the same `setFileContent()` path, so MD5-based change detection applies. The returned entity is then assigned to the new user and saved.

## Cloning via `ShareCloningService`

**File:** `src/services/share-cloning.service.ts`

When a user clones a shared project or component, the `ShareCloningService` creates a new `ProjectFile` entity and copies the content:

```typescript
cloned.elementsFile = new ProjectFile();
if (project.elementsFile)
    cloned.elementsFile.setFileContent(await project.elementsFile.getFileContent());
```

A few important things to note:

1. A **new** `ProjectFile` instance is created — the new entity gets its own UUID filename.
2. `getFileContent()` reads the source file from disk into memory.
3. `setFileContent()` feeds the same bytes into the new entity — this computes an MD5 hash (which will be the same as the source, since the content is identical).
4. When the cloned project is saved, `@AfterInsert()` writes the content to a new file under a new UUID filename.
5. The original file on disk is untouched — the clone is a full independent copy.

The same pattern applies to component cloning (the service clones the component with its dependencies, and each cloned component gets its own `ComponentFile`).

Note that the `ShareCloningService` operates inside a `@Transaction()`, so if the transaction rolls back, the `@AfterInsert` hook will have already written the file to disk. This means **orphan files** are possible on transaction failure. In practice, this is acceptable because:
- Transaction failures are rare (conflict-free writes to independent tables).
- Orphaned JSON files are small and do not accumulate quickly.
- There is no cleanup mechanism for orphaned files.

## The MD5 Hash Column

The `hash` column (type `char(32)`) stores the MD5 hex digest of the file content. Its purpose is **change detection, not security**.

- MD5 is used because it is fast — it is a non-cryptographic use case where speed matters more than collision resistance.
- The hash is computed on every call to `setFileContent()`.
- If the new hash matches the stored hash, `_dirty` stays `false` and the `@BeforeUpdate` hook skips the filesystem write entirely.
- On initial creation (when the row has not been persisted yet), `_dirty` will be `true` regardless because the entity starts with `_dirty = false` and no row exists, but TypeORM will fire `@AfterInsert` (not `@BeforeUpdate`) for new entities, so the check still works correctly.

The default value is `00000000000000000000000000000000` (32 zeros), which is a valid MD5 hash of a specific input. This is safe because:
- When a new entity is created, `setFileContent()` is called before insert, which overwrites the hash.
- If `setFileContent()` is never called (an empty file), the hash remains all zeros, which is the MD5 of an empty buffer — so it is actually correct.

## Non-Obvious Patterns

### `filePath` uses a relative walk from `__dirname`

```typescript
path.join(__dirname, '..', '..', '..', 'resources', this._path, ...)
```

This assumes the compiled output directory structure mirrors the source. If the build process changes the relative depth, the `..` segments would need updating. In practice, the TypeScript build keeps the same nesting (compiled `.js` files are at the same relative depth as the source `.ts` files), so this has been stable.

### `publicUrl` strips `public/` from the path

The `publicUrl` getter does `this._path.replace(/^public\//, '')` to produce a URL path that matches the Express static mount at `resources/public/`. If a subclass were to set `_path` without the `public/` prefix, the URL would be incorrect. This is an implicit convention: public resources must have `_path` starting with `public/`.

### ProjectFile and ComponentFile override `publicUrl` with `@Exclude()`

Even though the base class exposes `publicUrl` via `@Expose()`, the JSON file subclasses override the getter and decorate it with `@Exclude()` from `class-transformer`. This ensures circuit JSON file URLs are never leaked through the API. The JSON files are in the `private/` tree anyway, so their URLs would not work even if exposed.

### No explicit `@BeforeInsert` — only `@AfterInsert`

Note that there is no `@BeforeInsert` hook. The `@AfterInsert` hook is used to write the file after the row is created. This means if the insert fails (constraint violation, connection error), the file is never written — which is correct behavior. But it also means the hook runs **after** the transaction commits when used inside a `@Transaction()` method, which is why clones in `ShareCloningService` can leave orphan files on rollback (see above).

### The `_cacheable` flag is checked at update time, not insert

The cache-busting logic (delete old file + generate new UUID) only runs in `@BeforeUpdate`. On initial `@AfterInsert`, the file is simply written with whatever UUID was generated. This means if you insert a cacheable entity with content, then immediately call `setFileContent()` again with different content, the first file will be orphaned (since it was written by `@AfterInsert`, then `@BeforeUpdate` for the first update will delete it and write a new one). In practice, this does not happen because cacheable entities are always created empty and populated in the same code path.

### Entities are cascade-deleted from parent `@BeforeRemove` hooks

The `Project` entity has an explicit `@BeforeRemove` hook that deletes the `elementsFile` before the project itself is removed. This exists because the `cascade` option on the `@OneToOne` relation does not trigger TypeORM lifecycle hooks on the related entity — only the `Project` hook manually calls `remove()` on the file entity, which in turn fires `@BeforeRemove` on `PersistedResource` to delete the file from disk. The same pattern applies to `Component` and `User`.

```typescript
// Project.entity.ts — before remove
@BeforeRemove()
private async deleteFile() {
    if (this.elementsFile)
        await getCustomRepository(ProjectFileRepository).remove(this.elementsFile);
}
```

### Preview and profile pictures are `eager: true` relations

Both `Project` and `Component` load their preview entities eagerly (`{ eager: true }`). This means every project/component fetch automatically loads the associated preview entities and their `publicUrl` properties. This is intentional — the community gallery and search results need to display thumbnails on every page. However, it also means that the in-memory `_fileContent` of preview entities is never populated by the eager load (only the metadata columns are read from the database), so accessing `getFileContent()` on a preview entity requires a separate filesystem read.

### The `_fileContent` cache is per-instance

The in-memory `_fileContent` buffer is not shared between entity instances. If you fetch the same project file twice from the repository, you get two entity instances, each of which will read the file from disk independently on the first `getFileContent()` call. There is no cross-instance cache or LRU. For the expected workloads (one project open at a time), this is fine.
