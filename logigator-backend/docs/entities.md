# Entities

The entity layer models every persisted object in the system. Entities are TypeORM-decorated classes stored in `src/database/entities/`. They use `class-transformer` for serialization ‚Äî every response the API sends passes through `classToPlain()` (or the `@ResponseClassTransformOptions` decorator) to control which fields reach the client.

## Core Concepts

### UUID Primary Keys as the Norm

Every entity uses `@PrimaryGeneratedColumn('uuid')` for its `id` field. UUIDs are generated on the application side (not by the database), which makes them safe for distributed creation and avoids auto-increment enumeration attacks. The `link` field on `Project` and `Component` is also a UUID (`@Generated('uuid')`), generated by the database at insert time and used as the public share URL token.

### `@Exclude({toPlainOnly: true})` ‚Äî Defense-in-Depth Serialization

All entities except `Shortcut` carry the class-level decorator:

```ts
@Exclude({toPlainOnly: true})
```

This tells `class-transformer` to **exclude every property by default** during serialization (`toPlain`). Only properties explicitly annotated with `@Expose()` are included in the output. This is a defense-in-depth approach:

- It prevents accidental exposure of new columns added to an entity without serialization review.
- It prevents sensitive data like `password`, `_filename`, `_hash`, and internal `_dirty` flags from ever reaching the API response.
- The `{toPlainOnly: true}` scoping means the exclusion only applies when converting to plain objects (API responses), not when converting from plain objects to class instances (request deserialization).

`Shortcut` uses the shorter `@Exclude()` (which applies to both `toPlain` and `toClass`), because shortcuts are never deserialized from client input ‚Äî they are created programmatically from a typed DTO.

Selective `@Expose()` annotations then whitelist the fields that should appear in responses, often scoped to specific serialization groups (see below).

## Serialization Groups

API responses use `@ResponseClassTransformOptions({groups: [...]})` from `routing-controllers` to control which fields appear in different contexts. The groups form an additive whitelist ‚Äî a field exposed in group `'A'` appears in any serialization that includes group `'A'`.

### Default Group (no group specified)

Fields exposed without a group restriction appear in **every** response. These are the basic, non-sensitive fields:

- `id`, `name`, `description`, `createdOn`, `lastEdited`
- `previewDark`, `previewLight` (the preview entity objects, which themselves expose `publicUrl` and `hash`)
- `public` flag
- `symbol`, `numInputs`, `numOutputs`, `labels` (Component only)
- `username`, `memberSince`, `image` (User only)
- `name`, `keyCode`, `shift`, `ctrl`, `alt` (Shortcut only)
- `mimeType`, `hash`, `publicUrl` (PersistedResource subclasses)

### `'showShareLinks'`

Includes the `link` field on `Project` and `Component`. This group is used when the authenticated user is viewing their own resources ‚Äî the share link UUID is sensitive (whoever has it can view the resource) and should only be returned to the owner.

Used in:
- `ProjectController.list()` and `ProjectController.update()`
- `ComponentController.list()` and `ComponentController.create()`
- `ShareController.get()` and `ShareController.getDependencies()` (for public share pages)

### `'privateUserData'`

Used exclusively on `User` endpoints. Includes:

- `email` ‚Äî only the account owner should see their email.
- `shortcuts` ‚Äî the private proxy property `__shortcuts__`.
- `projects` / `components` ‚Äî the full list (including private ones), via private proxy properties `__projects__` / `__components__`.

Used in `UserController.get()` and `UserController.update()`.

### `'extendedUserData'`

Used when another user views a profile. Includes `projects` and `components`, but the accompanying `@Transform` filter strips out non-public entries:

```ts
@Transform(({value}) => {
    return value ?? value.filter(p => p.public);
}, {groups: ['extendedUserData']})
```

The `value ??` fallback ensures that if the proxy property was never populated (i.e., the lazy relation was never awaited), serialization returns `undefined` rather than crashing.

### `'detailed'`

Includes nested relational data beyond what the default group provides:

- `user` ‚Äî the owning User object (full entity, which itself is then filtered by User's own `@Exclude`). Used when detailed owner information is needed.
- `forks` ‚Äî the list of child projects/components that forked from this one.
- `dependencies` / `dependencyForComponents` / `dependencyForProjects` ‚Äî the dependency join entities with their component references.
- `forkedFrom` (already exposed in the default group ‚Äî see below).

### `'compactDependencies'`

A specialized transform group on `ProjectDependency` and `ComponentDependency`. Instead of serializing the full `dependency` component object, it transforms it to just the component ID:

```ts
@Transform(({value}) => value.id, {groups: ['compactDependencies']})
```

This is used when a lightweight dependency list is sufficient (e.g., for tree-shaking dependency graphs without loading full component data).

## Private Proxy Property Pattern

TypeORM lazy relations are declared as `Promise<T>`:

```ts
@ManyToOne(() => User, object => object.projects, {nullable: false, onDelete: 'CASCADE'})
user: Promise<User>;
```

The `Promise` wrapper is asynchronous ‚Äî TypeORM resolves it on first property access, but `class-transformer` cannot `await` promises during serialization. The promise object itself is not serializable, and the resolved value is not available synchronously.

The solution is a **private proxy property** pattern:

```ts
@Expose({name: 'user', groups: ['detailed']})
private __user__: User;
```

Controllers manually `await` the lazy relation and assign the result to the private proxy before the entity reaches the serializer:

```ts
// In a controller:
const project = await this.projectRepo.findOne(...);
project.__user__ = await project.user;  // populate proxy
return classToPlain(project, {groups: ['detailed']});
```

The `@Expose({name: 'user'})` tells `class-transformer` to serialize `__user__` under the key `"user"`, matching the original property name.

This pattern appears in every entity that has lazy `Promise<T>` relations:

| Entity | Lazy Relation | Proxy Property | Groups |
|--------|--------------|----------------|--------|
| Project | `user: Promise<User>` | `__user__` | `detailed` |
| Project | `forkedFrom: Promise<Project>` | `__forkedFrom__` | _none (default)_ |
| Project | `forks: Promise<Project[]>` | `__forks__` | `detailed` |
| Project | `dependencies: Promise<ProjectDependency[]>` | `__dependencies__` | `detailed` |
| Component | `user: Promise<User>` | `__user__` | `detailed` |
| Component | `forkedFrom: Promise<Component>` | `__forkedFrom__` | _none (default)_ |
| Component | `forks: Promise<Component[]>` | `__forks__` | `detailed` |
| Component | `dependencies: Promise<ComponentDependency[]>` | `__dependencies__` | `detailed` |
| Component | `dependencyForComponents: Promise<ComponentDependency[]>` | `__dependencyForComponents__` | `detailed` |
| Component | `dependencyForProjects: Promise<ProjectDependency[]>` | `__dependencyForProjects__` | `detailed` |
| User | `shortcuts: Promise<Shortcut[]>` | `__shortcuts__` | `privateUserData` |
| User | `projects: Promise<Project[]>` | `__projects__` | `privateUserData`, `extendedUserData` |
| User | `components: Promise<Component[]>` | `__components__` | `privateUserData`, `extendedUserData` |

Note that `forkedFrom` is exposed in the **default** (no-group) serialization, so it always appears when available. The `user` relation is only exposed in the `'detailed'` group, requiring an explicit opt-in.

Not all controllers fully populate every proxy ‚Äî the pattern is only used where the serialization group demands it. For example, `ProjectController.open()` does NOT set `__user__` or `__forks__` because it only uses `'showShareLinks'` group, which doesn't include those relations.

## TypeORM Lazy Relations (All `Promise<T>`)

Every to-one and to-many relationship in the entity layer uses TypeORM lazy loading ‚Äî the property type is `Promise<T>` rather than `T`. This means:

- The relation is not resolved until the property is `await`ed (or `.then()` is called).
- The promise is cached by TypeORM after the first resolution; subsequent accesses return the same resolved value.
- Relations that are **not** eagerly loaded (via `eager: true`) must be explicitly awaited before use.

The one exception to lazy loading is the `elementsFile` relation on `Project` and `Component`, which uses `eager: true`:

```ts
@OneToOne(() => ProjectFile, projectFile => projectFile.project, {cascade: true, eager: true})
elementsFile: ProjectFile;
```

This means `elementsFile` (and its `PersistedResource` subclass) is always loaded with the parent entity ‚Äî no separate query needed. The same applies to `previewDark`, `previewLight`, and `image` (User's `ProfilePicture`).

## Entity Hierarchy and Relationships

### User (`user.entity.ts`)

Represents a registered user account with multiple login strategies.

**Login strategies** (mutually exclusive, enforced by a `@Check` constraint):

| Strategy | Column | Type | Notes |
|----------|--------|------|-------|
| Local | `password` | `string \| null` | Hashed with bcrypt (9 rounds). Requires `localEmailVerified = true`. |
| Google | `googleUserId` | `string \| null` | Unique. OAuth subject ID from Google. |
| Twitter | `twitterUserId` | `string \| null` | Unique. OAuth subject ID from Twitter. |

The `@Check` constraint `(login_type = 'local' and password is not null) or (login_type != 'local')` enforces at the database level that local accounts always have a password. Note that `login_type` is not a column in the entity definition ‚Äî it is a computed column or view column created by a database migration; the check constraint references it as it exists in the database schema.

**Relationships:**

```ts
@OneToOne(() => ProfilePicture, image => image.user, {cascade: true, eager: true})
image: ProfilePicture;          // Eager ‚Äî always loaded with User.

@OneToMany(() => Shortcut, object => object.user, {cascade: true})
shortcuts: Promise<Shortcut[]>; // Cascade ‚Äî deleting User deletes all Shortcuts.

@OneToMany(() => Project, object => object.user, {cascade: true})
projects: Promise<Project[]>;   // Cascade ‚Äî deleting User deletes all Projects.

@OneToMany(() => Component, object => object.user, {cascade: true})
components: Promise<Component[]>; // Cascade ‚Äî deleting User deletes all Components.

@ManyToMany(() => Project, project => project.stargazers)
staredProjects: Promise<Project[]>; // Starred projects (not owned).

@ManyToMany(() => Component, component => component.stargazers)
staredComponents: Promise<Component[]>; // Starred components (not owned).
```

**`@BeforeRemove` cleanup:**

When a `User` is removed, `@BeforeRemove` cascades deletion to all owned content by using custom repositories (not TypeORM cascade, because the file-backed entities need lifecycle hooks to run):

1. Fetches all projects and removes them via `ProjectRepository` (triggering `Project.@BeforeRemove` which deletes `ProjectFile`).
2. Fetches all components and removes them via `ComponentRepository` (triggering `Component.@BeforeRemove` which deletes `ComponentFile`).
3. Removes the profile picture via `ProfilePictureRepository` if it exists.

This is necessary because TypeORM's native cascade does not guarantee that lifecycle hooks on indirectly-removed entities fire correctly when the parent is deleted.

### Project / Component ‚Äî Nearly Identical Structure

`Project` and `Component` share an almost identical schema with the same ownership, file, preview, fork, and dependency semantics:

| Field | Project | Component |
|-------|---------|-----------|
| `name` | `varchar(20)` | `varchar(20)` |
| `description` | `varchar(2048)` | `varchar(2048)` |
| `createdOn` | `datetime` (automatic) | `datetime` (automatic) |
| `lastEdited` | `datetime` (automatic) | `datetime` (automatic) |
| `elementsFile` | `ProjectFile` (eager) | `ComponentFile` (eager) |
| `link` | UUID (for sharing) | UUID (for sharing) |
| `public` | boolean | boolean |
| `forkedFrom` | `Promise<Project>` | `Promise<Component>` |
| `forks` | `Promise<Project[]>` | `Promise<Component[]>` |
| `stargazers` | `Promise<User[]>` (ManyToMany) | `Promise<User[]>` (ManyToMany) |
| `stargazersCount` | number (plain, loaded by query) | number (plain, loaded by query) |
| ‚Äî | ‚Äî | `symbol: varchar(5)` |
| ‚Äî | ‚Äî | `numInputs: number` |
| ‚Äî | ‚Äî | `numOutputs: number` |
| ‚Äî | ‚Äî | `labels: string[]` (simple-array) |

**Component-specific fields:**

- `symbol` ‚Äî a short string (max 5 characters) representing the component type in the circuit editor (e.g., `'AND'`, `'XOR'`).
- `numInputs` / `numOutputs` ‚Äî port counts for the sub-circuit.
- `labels` ‚Äî an array of port label strings, stored as a TypeORM `simple-array` (comma-separated in the database; careful with labels containing commas).

**Dependency direction ‚Äî Project vs Component:**

A `Project` can depend on `Component` entities (via `ProjectDependency`). A `Component` can depend on other `Component` entities (via `ComponentDependency`), forming a directed acyclic dependency graph. Projects cannot depend on other projects.

**`@BeforeRemove` cleanup:**

Both entities delete their `elementsFile` (the `ProjectFile` or `ComponentFile`) before removal, so the associated file on disk is cleaned up:

```ts
@BeforeRemove()
private async removeFile() {
    if (this.elementsFile)
        await getCustomRepository(ProjectFileRepository).remove(this.elementsFile);
}
```

Deletion of `Project` / `Component` also cascades to their `ProjectDependency` / `ComponentDependency` join entities via the `onDelete: 'CASCADE'` foreign key constraint (not via lifecycle hooks).

### PersistedResource (`persisted-resource.entity.ts`) ‚Äî Abstract Base for File-Backed Entities

`PersistedResource` provides a generic file-storage layer. Every entity that needs to store a binary blob on disk (circuit JSON, preview images, profile pictures) extends this abstract class.

**Class-level `@Exclude`:**

The base class uses `@Exclude({toPlainOnly: true})`. Subclasses inherit this exclusion ‚Äî they must explicitly `@Expose()` any field they want to appear in API responses.

**Columns:**

| Column | Type | Notes |
|--------|------|-------|
| `id` | `uuid` (PK) | Primary key, generated by the application. |
| `_filename` | `uuid` (unique, generated) | The actual filename on disk (no extension). Generated automatically by TypeORM's `@Generated('uuid')`. Private via leading underscore ‚Äî not serialized. |
| `mimeType` | `varchar` | `@Expose()` ‚Äî reported to clients for Content-Type headers. |
| `_hash` | `char(32)` | MD5 hex digest of `_fileContent`. Private ‚Äî used by clients for cache validation (`oldHash` in save requests). Exposed via the `hash` getter. |

**`_path` and `_cacheable` ‚Äî subclass configuration:**

Subclasses set `protected _path` to determine the filesystem subdirectory and `protected _cacheable` to control filename regeneration:

| Subclass | `_path` | `_cacheable` | File System Location |
|----------|---------|-------------|---------------------|
| `ProjectFile` | `private/projects` | `false` | `resources/private/projects/{uuid}.json` |
| `ComponentFile` | `private/components` | `false` | `resources/private/components/{uuid}.json` |
| `ProjectPreviewDark` | `public/preview/dark` | `true` | `resources/public/preview/dark/{uuid}.png` |
| `ProjectPreviewLight` | `public/preview/light` | `true` | `resources/public/preview/light/{uuid}.png` |
| `ComponentPreviewDark` | `public/preview/dark` | `true` | `resources/public/preview/dark/{uuid}.png` |
| `ComponentPreviewLight` | `public/preview/light` | `true` | `resources/public/preview/light/{uuid}.png` |
| `ProfilePicture` | `public/profile` | `true` | `resources/public/profile/{uuid}.png` |

**`filePath` getter:**

```ts
public get filePath(): string {
    return path.join(__dirname, '..', '..', '..', 'resources', this._path,
        this._filename + '.' + mime.getExtension(this.mimeType));
}
```

Resolves relative to `logigator-backend/src/database/entities/`, traversing up three directories to reach the project root, then into `resources/{_path}/{filename}.{ext}`.

**`publicUrl` getter:**

```ts
@Expose()
public get publicUrl(): string {
    return `/${this._path.replace(/^public\//, '')}/${this._filename}.${mime.getExtension(this.mimeType)}`;
}
```

Strips the `public/` prefix from the path. For example, a profile picture with `_path = 'public/profile'` produces `/profile/{uuid}.png`. This URL path is served statically by the web server (Apache / nginx).

**`publicUrl` override ‚Äî excluding private files:**

`ProjectFile` and `ComponentFile` override `publicUrl` to exclude it from serialization:

```ts
@Exclude()
public get publicUrl() {
    return super.publicUrl;
}
```

This is critical ‚Äî circuit data files live under `private/projects` and `private/components` and must never have publicly accessible URLs. The override re-decorates the getter with `@Exclude()` to prevent accidental exposure.

**`getFileContent()` ‚Äî lazy caching:**

```ts
public async getFileContent(): Promise<Buffer> {
    if (!this._fileContent)
        this._fileContent = await fs.readFile(this.filePath);
    return this._fileContent;
}
```

Reads the file from disk on first access and caches the result in the `_fileContent` property. Subsequent calls return the cached buffer. This is the primary way controllers access the circuit JSON data:

```ts
const contentBuffer = await project.elementsFile?.getFileContent();
const content = contentBuffer?.length ? JSON.parse(contentBuffer.toString()) : [];
```

**`setFileContent()` ‚Äî dirty tracking and MD5 hashing:**

```ts
public setFileContent(content: Buffer | string) {
    // ...store content, compute hash, set _dirty flag
}
```

Computes an MD5 hash of the new content and compares it to the stored `_hash`. Sets `_dirty = true` only if the hash changed. The hash is used for optimistic concurrency control ‚Äî clients send the old hash with save requests, and the server rejects updates if they don't match:

```ts
if (project.elementsFile && project.elementsFile.hash !== body.oldHash)
    throw new BadRequestError('VersionMismatch');
```

**`_cacheable` ‚Äî cache-busting via filename rotation:**

When `_cacheable = true` (previews, profile pictures), the `@BeforeUpdate` lifecycle hook generates a **new UUID filename** whenever the content changes:

```ts
@BeforeUpdate()
private async updateFile(): Promise<void> {
    if (this._dirty) {
        if (this._cacheable) {
            await this.deleteFile();
            this._filename = uuid();    // <-- new filename = cache bust
        }
        await fs.writeFile(this.filePath, this._fileContent);
        this._dirty = false;
    }
}
```

The old file is deleted and a new one is created with a fresh UUID. This means the `publicUrl` changes on every update, forcing CDN and browser caches to refetch the resource. This is appropriate for preview images and profile pictures where cache invalidation is important.

When `_cacheable = false` (project/component files), the filename stays the same and only the content is overwritten. These files are not served publicly, so cache-busting is unnecessary.

**Lifecycle hooks summary:**

| Decorator | Method | Action |
|-----------|--------|--------|
| `@AfterInsert()` | `createFile()` | Writes `_fileContent` to disk at `filePath`. Creates an empty buffer if no content was set. |
| `@BeforeUpdate()` | `updateFile()` | If dirty: optionally regenerates filename (cacheable), writes content to disk, clears dirty flag. |
| `@BeforeRemove()` | `deleteFile()` | Unlinks the file from disk. Silently ignores `ENOENT` (file already gone). |

### ProfilePicture (`profile-picture.entity.ts`)

Extends `PersistedResource`:

```ts
protected _path = 'public/profile';
protected _cacheable = true;
```

Linked to `User` via a bidirectional `@OneToOne`:

```ts
@OneToOne(() => User, user => user.image, {onDelete: 'CASCADE', onUpdate: 'CASCADE'})
@JoinColumn()
user: Promise<User>;
```

The `ProfilePictureRepository` provides an `importFromUrl(url)` method that fetches the image from a remote URL (used during OAuth login to import Google/Twitter profile photos), saves it to the filesystem, and returns the entity.

### ProjectFile / ComponentFile (`project-file.entity.ts`, `component-file.entity.ts`)

Store the circuit JSON data (serialized array of `ProjectElement` objects). Both extend `PersistedResource` with:

```ts
public mimeType = 'application/json';
protected _path = 'private/projects';    // or 'private/components'
```

The `publicUrl` getter is overridden with `@Exclude()` to prevent exposure.

The owning side of the `@OneToOne` relationship is on the `File` entity (it has `@JoinColumn()`), while the inverse side is on `Project` / `Component`:

```ts
// ProjectFile
@OneToOne(() => Project, project => project.elementsFile, {onDelete: 'CASCADE', onUpdate: 'CASCADE'})
@JoinColumn()
project: Promise<Project>;
```

### Preview Entities (`project-preview-dark.entity.ts`, `project-preview-light.entity.ts`, `component-preview-dark.entity.ts`, `component-preview-light.entity.ts`)

Four nearly identical entities storing PNG preview images of circuits. Each pairs a direction (dark/light theme) with an owner type (project/component):

| Entity | `_path` | `_cacheable` |
|--------|---------|-------------|
| `ProjectPreviewDark` | `public/preview/dark` | `true` |
| `ProjectPreviewLight` | `public/preview/light` | `true` |
| `ComponentPreviewDark` | `public/preview/dark` | `true` |
| `ComponentPreviewLight` | `public/preview/light` | `true` |

All four have `_cacheable = true`, so updating a preview creates a new filename and cache-busts the old URL.

**Why previews are separate entities, not columns on Project/Component:**

Previews are stored as files on disk (PNG images), not as binary database columns. Each preview extends `PersistedResource`, which gives it:

- File lifecycle hooks (`@AfterInsert` writes the file, `@BeforeUpdate` updates it, `@BeforeRemove` deletes it).
- MD5 hashing for change detection.
- Cache-busting via filename rotation (`_cacheable`).
- Efficient serving via the web server's static file handler (no Express route needed).

Storing previews as columns (e.g., `BLOB`) would put binary data into the database, increasing backup size, slowing down queries that select project/component rows, and preventing direct file serving by the web server.

### ProjectDependency / ComponentDependency ‚Äî Composite Primary Key Join Entities

These entities model a many-to-many relationship with additional data (the `model_id` column). They use a **composite primary key** formed by two `@ManyToOne` columns, each decorated with `{primary: true}`:

```ts
// ProjectDependency
@Entity()
export class ProjectDependency {
    @ManyToOne(() => Project, object => object.dependencies, {
        primary: true,    // <-- part of composite PK
        eager: true,
        nullable: false,
        onDelete: 'CASCADE'
    })
    dependent: Project;

    @Expose()
    @Transform(({value}) => value.id, {groups: ['compactDependencies']})
    @ManyToOne(() => Component, object => object.dependencyForProjects, {
        primary: true,    // <-- part of composite PK
        eager: true,
        nullable: false,
        onDelete: 'CASCADE'
    })
    dependency: Component;

    @Expose({name: 'model'})
    @Column({nullable: false})
    model_id: number;
}
```

Both relations are `eager: true`, so a dependency join entity always carries its component reference. The `model_id` column maps the dependency to a specific model slot in the parent circuit.

`ComponentDependency` follows the same pattern, but both sides point to `Component`:

| Entity | `dependent` type | `dependency` type | `model_id` |
|--------|-----------------|-------------------|------------|
| `ProjectDependency` | `Project` | `Component` | number |
| `ComponentDependency` | `Component` | `Component` | number |

The `'compactDependencies'` group transform replaces the full dependency entity with just its component ID:

```ts
@Transform(({value}) => value.id, {groups: ['compactDependencies']})
```

**Dependency graph semantics:**

- `ComponentDependencyRepository.getDependencies(component, recursive)` fetches the full dependency tree recursively. This is used when cloning a component ‚Äî all transitive dependencies must also be cloned.
- `ProjectDependencyRepository.getDependencies(project, recursive)` first fetches the project's direct component dependencies, then recursively fetches those components' own dependencies.
- The `ShareCloningService` uses these recursive lookups in a `@Transaction()`-wrapped method to atomically clone the entire dependency graph.

### Shortcut (`shortcut.entity.ts`)

Per-user keyboard shortcut bindings. `@Exclude()` (without `{toPlainOnly: true}`) at the class level means shortcuts are excluded from both serialization and deserialization ‚Äî they are created programmatically from typed DTOs, never from raw client JSON.

```ts
@Entity()
@Unique(['name', 'user'])
export class Shortcut {
    @PrimaryGeneratedColumn('uuid')
    id: string;

    @Expose()
    @Column({nullable: false})
    name: string;

    @Expose()
    @Column({nullable: false})
    keyCode: string;

    @Expose()
    @Column({default: false, nullable: false})
    shift: boolean;

    @Expose()
    @Column({default: false, nullable: false})
    ctrl: boolean;

    @Expose()
    @Column({default: false, nullable: false})
    alt: boolean;

    @ManyToOne(() => User, object => object.shortcuts, {nullable: false, onDelete: 'CASCADE'})
    user: Promise<User>;
}
```

The `@Unique(['name', 'user'])` constraint ensures that each user has at most one binding for a given action name. When a user updates their shortcuts, the controller replaces all existing shortcuts atomically:

```ts
await this.shortcutRepo.remove(await this.shortcutRepo.find({ where: { user } }));
user.shortcuts = Promise.resolve(body.shortcuts.map(x => x.toEntity()));
await this.userRepo.save(user);
```

## `@Transaction()` and `@TransactionRepository()` Decorator Pattern

The `@Transaction()` decorator (from TypeORM) wraps a method in a database transaction. Injected `@TransactionRepository()` parameters provide repository instances scoped to that transaction, ensuring all operations within the method run in the same database connection and can be rolled back atomically.

This is used in `ShareCloningService` for the clone operations:

```ts
@Transaction()
private async cloneProjectTransaction(
    project: Project,
    dependencies: Component[],
    user: User,
    @TransactionRepository(ComponentRepository) compRepo?: ComponentRepository,
    @TransactionRepository(ProjectRepository) projRepo?: ProjectRepository,
    @TransactionRepository(ComponentDependencyRepository) compDepRepo?: ComponentDependencyRepository,
    @TransactionRepository(ProjectDependencyRepository) projDepRepo?: ProjectDependencyRepository
) {
    // All repo operations here share a single transaction
}
```

And in `UserService.removeTransaction()` for account deletion:

```ts
@Transaction()
private async removeTransaction(
    user: User,
    @TransactionRepository(ComponentRepository) compRepo?: ComponentRepository,
    @TransactionRepository(ProjectRepository) projRepo?: ProjectRepository,
    @TransactionRepository(UserRepository) userRepo?: UserRepository,
    @TransactionRepository(ProfilePictureRepository) profilePicRepo?: ProfilePictureRepository
) {
    // Delete all components, projects, and profile picture in one transaction
}
```

**Pattern details:**

- The method must be `private` or `protected` (TypeORM limitation ‚Äî public transaction methods have issues with proxies).
- Repository parameters are optional (`?`) because they are injected by TypeORM at runtime, not passed by the caller.
- All repositories used inside the transaction must be injected via `@TransactionRepository()` ‚Äî using repository instances from the constructor would run outside the transaction.

## Non-Obvious Patterns

### Previews as Separate Entities (Not Columns)

Circuit previews (dark/light PNG images) are stored as separate `PersistedResource` subclasses rather than as binary columns on `Project` / `Component`. Reasons:

1. **File lifecycle management** ‚Äî Each preview has its own `@AfterInsert` / `@BeforeUpdate` / `@BeforeRemove` hooks that manage the file on disk. Adding `BLOB` columns to `Project` would require the application to handle file I/O manually.
2. **Cache-busting** ‚Äî Preview entities have `_cacheable = true`, which rotates the filename on every update. This cannot work with in-database blobs.
3. **Static file serving** ‚Äî Previews are served directly by the web server from `resources/public/preview/`, bypassing Express entirely. Binary columns would require an Express route to serve them.
4. **Query efficiency** ‚Äî Loading `Project` rows does not load binary image data; preview entities are loaded only when explicitly joined.

### Cache-Busting Filenames

For public, cacheable resources (previews, profile pictures), the filename is a UUID that changes every time the content is updated:

```
Update #1: /preview/dark/a1b2c3d4-....png
Update #2: /preview/dark/e5f6g7h8-....png  (different filename)
```

Since the `publicUrl` changes, any CDN, browser cache, or reverse proxy that cached the old URL will miss on the new one and fetch fresh content. The old file is deleted by `@BeforeUpdate`.

### Stargazers Count

The `stargazersCount` property on `Project` and `Component` is **not a database column** ‚Äî it is a plain number field that is populated at query time by `loadRelationCountAndMap`:

```ts
// In ProjectRepository.getProjectWithStargazersCountByLink():
return await this.createQueryBuilder('project')
    .loadRelationCountAndMap('project.stargazersCount', 'project.stargazers')
    .leftJoinAndSelect('project.previewDark', 'previewDark')
    .leftJoinAndSelect('project.previewLight', 'previewLight')
    .where('project.link = :link', {link})
    .getOne();
```

When using a paginated shared listing, stargazers count is computed differently ‚Äî via a GROUP BY/COUNT subquery:

```ts
.addSelect('COUNT(stargazers.id)', 'stargazersCount')
// ...
.groupBy('project.id')
.orderBy('stargazersCount', 'DESC')
```

This means `stargazersCount` is `undefined` on entities loaded without one of these specific query builders.

### The `link` Field ‚Äî Public Share Token

Every `Project` and `Component` has a `link` column (`UUID`, `@Generated('uuid')`) that serves as the public share URL token. It is generated by the database at insert time (not by the application), ensuring uniqueness without a round-trip.

The link is `@Expose({groups: ['showShareLinks']})` ‚Äî only returned to the authenticated owner of the resource. Other users access shared resources by knowing the link (it's part of the share URL), but the API does not return it in public listings.

Regenerating the link is a deliberate user action:

```ts
// In ProjectController.update():
if (body.updateLink)
    project.link = uuid();   // application-generated new UUID
```

### Service Layer Separation

Entity classes contain no business logic ‚Äî they are data models with lifecycle hooks only. Business logic lives in `src/services/`:

- `UserService` ‚Äî user CRUD, password hashing, OAuth login, email verification.
- `ShareCloningService` ‚Äî cloning projects/components with all dependencies (uses `@Transaction()`).
- `StandaloneViewService` ‚Äî rendering Handlebars templates for emails.

Services use `@InjectRepository()` from `typeorm-typedi-extensions` to inject repository instances, which themselves extend TypeORM's `Repository` or `PageableRepository`.
